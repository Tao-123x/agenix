# V0.3 Customer Acceptance: Real Adapter Path

## Reviewer

Maya Chen, first-customer representative for Agenix.

## Document Status

This is a customer acceptance standard, not release notes.

It does not claim that V0.3 functionality is already implemented. It describes
what I would need to see before I accept V0.3 as a credible step from the V0.2
authoring release toward V1.

## Customer Problem

V0.2 proves that Agenix can package, check, run, verify, and replay authored
skills through a local reference-runtime loop.

That is useful, but it is not enough for a real platform team moving toward V1.
The next risk is adapter trust. A real user needs to know:

- which adapters exist and what each one is allowed to do
- whether a skill and adapter are compatible before execution starts
- whether a provider-backed adapter can be smoke-tested without becoming the
  default execution path
- why a failed run failed, in a way CI and humans can both act on
- whether trace evidence is specific enough to audit adapter selection,
  preflight, execution, verifier results, and final status

If those questions require reading runtime source code, V0.3 is not acceptable.

## Customer Position

I would accept V0.3 as a bridge release only if it keeps the V0.2 local authoring
contract intact while making the adapter boundary explicit enough for real
trial users.

I would not treat V0.3 as a procurement-ready V1. The goal is to prove the
adapter surface, not to expand the trust model.

## Required Regression Baseline

V0.3 must not weaken the V0.2 acceptance story.

The existing local release gates should still pass before any adapter-specific
acceptance is considered:

```bash
go run ./cmd/agenix acceptance
go run ./cmd/agenix acceptance --v0.2
go run ./cmd/agenix acceptance --v0.3
go test -count=1 ./...
go vet ./...
go build ./cmd/agenix
```

The exact V0.3 acceptance command is now `agenix acceptance --v0.3`. It must
remain documented and repeatable. A user should not have to reconstruct the
acceptance sequence from scattered examples.

The release workflow must run the offline gate on Linux, macOS, and Windows.
This is the evidence path for portability; a green local macOS run is not enough
to call the cross-OS requirement done.

## Adapter Catalog Acceptance

V0.3 needs a visible adapter catalog. This can be CLI output, generated docs, or
both, but the catalog must be user-facing.

For every adapter, the catalog must show:

- the exact adapter name a user passes to the CLI
- transport type, such as local builtin, local subprocess, or provider-backed
  remote
- supported skill IDs or supported skill classes
- declared capabilities, including tool calling, structured output, context
  limits, and reasoning level when relevant
- policy requirements, especially filesystem, shell, and network requirements
- whether the adapter is included in offline acceptance by default
- whether the adapter has a deterministic test, stub-provider test, or optional
  real-provider smoke path
- expected top-level failure classes for selection, preflight, execution,
  timeout, and verification failures
- expected trace event stages for selection, preflight, execution, and final
  status

This matters because a platform user should be able to answer "can this adapter
run this skill in CI?" without reading Go code or guessing from examples.

The catalog must not imply marketplace trust, publisher identity, signatures, or
security certification. It is a compatibility and behavior catalog, not a trust
store.

## Compatibility Preflight Acceptance

V0.3 must make compatibility preflight observable before meaningful execution.

At minimum, preflight must check:

- the selected adapter exists
- the adapter supports the requested skill
- the adapter satisfies `capabilities.requires`
- remote adapters are blocked unless the manifest and runtime policy explicitly
  allow network use
- unsupported or incompatible adapters fail before tool calls, filesystem
  writes, verifier execution, or provider requests

The failure surface must be stable:

- unknown adapter, unsupported skill, or capability mismatch should fail as
  `UnsupportedAdapter`
- malformed manifest or malformed trace should fail as `InvalidInput`
- network policy denial should fail as a policy error before any provider
  request is sent
- adapter execution failure after successful selection and preflight should
  remain distinct from verifier failure

The user-facing report must identify:

- selected adapter
- requested skill
- failed preflight stage
- missing or incompatible requirement
- actual adapter capability or policy state when available
- next useful recovery step, such as selecting another adapter, changing the
  manifest, or enabling an explicit smoke path

The machine-readable report must be schema-backed. A user should be able to run
`agenix validate` against a preflight JSON report and get a stable
`kind=adapter_compatibility_report` result before wiring it into CI.

This matters because adapter mismatch is a configuration problem, not an agent
reasoning problem. Users need to fix it before spending provider calls or
debugging irrelevant verifier output.

## Optional Real Provider Smoke Acceptance

V0.3 may include a real provider-backed adapter smoke path, but it must be
explicitly opt-in.

The smoke path is acceptable only if:

- default CI and default acceptance remain offline and do not require provider
  credentials
- the user must explicitly select the provider-backed adapter
- the manifest must explicitly allow the required network behavior
- missing credentials produce a clear failure report and do not look like a
  verifier failure
- provider requests have bounded timeout and response-size behavior
- read-only smoke is preferred first, so a provider-backed adapter can prove the
  boundary without gaining write authority
- trace records provider and model metadata only when safe to persist
- trace never persists API keys, authorization headers, raw provider request
  bodies, or raw provider response bodies

Manual smoke evidence should include:

- the exact command used
- the required environment variables
- the expected success shape
- expected failure examples for missing credentials, unauthorized credentials,
  rate limits, timeouts, oversized responses, and malformed provider responses
- the produced trace path
- confirmation that replay and verify still operate on the resulting trace
  according to the implemented trace contract

This matters because real-provider coverage is valuable only when users can run
it deliberately. It must not silently turn local Agenix usage into remote model
execution.

## Failure Report Acceptance

V0.3 failure reports must be useful for both humans and automation.

For CLI and JSON output, a failed run should expose:

- `error_class`
- `error_message`
- selected adapter
- requested skill
- failed stage, such as adapter selection, capability preflight, policy
  preflight, adapter execution, verifier execution, trace validation, or replay
- `trace_path` when a trace exists
- non-zero process exit
- a concise recovery hint

The failure report must preserve the existing taxonomy:

- `InvalidInput` for malformed manifests, malformed traces, and CLI usage
  errors
- `UnsupportedAdapter` for adapter selection and compatibility preflight
  failures
- `DriverError` for adapter execution failures after selection and preflight
  have passed
- `Timeout` for provider HTTP timeout behavior when that path is implemented
- `VerificationFailed` for verifier failure after adapter execution

This matters because CI needs to decide whether to retry, fail fast, ask for a
different adapter, fix a manifest, or inspect verifier output. A generic
"adapter failed" message is not enough.

## Trace Acceptance

V0.3 traces must make the adapter lifecycle auditable.

Every relevant run should let a reviewer see:

- `run_id`
- `skill`
- `model_profile`
- applied policy
- ordered events
- final status
- adapter selection result
- capability preflight result
- policy preflight result when network or other adapter policy matters
- adapter execution success or failure
- verifier results
- failure details that align with the user-facing failure report

Provider-backed traces must preserve audit context without preserving secrets.
The trace may include provider name, model name, configured timeout, and bounded
response metadata. It must not include credentials or raw provider payloads.

`verify` and `replay` must reject malformed traces with stable invalid-input
behavior. If replay remains trace-driven rather than deterministic
re-execution, the documentation must say that plainly.

This matters because trace is the only durable evidence a platform team can hand
to security, CI owners, and skill authors after a run finishes.

## Explicit Non-Scope For V0.3

These are not V0.3 acceptance requirements and should not be claimed as done by
this document:

- strong sandbox
- marketplace
- default remote execution
- signatures

It is acceptable for V0.3 documentation to say these are still outside the
current release boundary. It is not acceptable to imply they exist because a
provider-backed smoke path exists.

## Acceptance Checklist

I would accept V0.3 when the project can provide concrete evidence for all of
the following:

- V0.2 local acceptance still passes
- V0.3 offline acceptance passes on Linux, macOS, and Windows in CI
- adapter catalog is visible to users and explains local, builtin, and
  provider-backed adapter behavior without source-code reading
- compatibility preflight rejects unsupported adapter choices before execution
- preflight failures and execution failures use distinct stable error classes
- optional real-provider smoke is documented, credential-gated, and excluded
  from default offline acceptance
- failure reports include adapter, stage, error class, recovery hint, and trace
  path when available
- traces contain adapter selection, preflight, execution, verifier, and final
  status evidence
- provider-backed trace data is redacted and bounded
- docs explicitly state that strong sandbox, marketplace, default remote
  execution, and signatures remain out of scope

## Buyer Summary

V0.3 is acceptable only if it turns adapter use from an implementation detail
into a user-verifiable contract. That is the bridge from V0.2 authoring toward
V1 platform trust.

# Agenix v0.2.0 Plan

V0.2 is the Skill Authoring release.

The goal is to make the first reusable path for people who are not editing the
canonical demo by hand:

- initialize a skill from a template
- validate the generated manifest
- build a portable `.agenix` artifact
- run it through an explicit local adapter
- pass verifier before claiming success

First increment:

```bash
go run ./cmd/agenix init templates
go run ./cmd/agenix init templates --json
go run ./cmd/agenix init skill repo.demo_skill --template python-pytest -o /tmp/repo.demo_skill
go run ./cmd/agenix validate /tmp/repo.demo_skill/manifest.yaml
go run ./cmd/agenix build /tmp/repo.demo_skill -o /tmp/repo.demo_skill.agenix
go run ./cmd/agenix run /tmp/repo.demo_skill.agenix --adapter python-pytest-template
go run ./cmd/agenix check /tmp/repo.demo_skill --adapter python-pytest-template
go run ./cmd/agenix check /tmp/repo.demo_skill --adapter python-pytest-template --json > /tmp/report.json
go run ./cmd/agenix validate /tmp/report.json
```

Writable repair template:

```bash
go run ./cmd/agenix init skill repo.demo_fix --template repo-fix-test-failure -o /tmp/repo.demo_fix
python3 -m pytest -q /tmp/repo.demo_fix/fixture
go run ./cmd/agenix check /tmp/repo.demo_fix --adapter repo-fix-test-failure-template --json > /tmp/fix-report.json
go run ./cmd/agenix validate /tmp/fix-report.json
```

This does not widen V0.1 runtime trust boundaries. The template adapter is local
and deterministic, and generated skills still succeed only after verifiers pass.

`agenix check` is the authoring release gate for a single skill. It validates
the manifest, builds a temporary artifact, runs that artifact, validates the
trace, reruns verifier, and replays the trace summary. `--json` prints the same
gate result as a stable `kind: check_report` report for CI and agent-to-agent
consumption. The report is covered by `specs/check-report.schema.json` and can
be checked with `agenix validate`.

# Agenix v0.3.0 Acceleration Plan

V0.3 is the Model Adapter Readiness release.

Status: active target after V0.2.0. The first adapter discovery and compatibility
preflight increment has landed; the rest of this document remains a target plan
until the V0.3 release gate exists.

The release goal is to make real model adapters usable without weakening the
runtime contract. "Ready" means a developer can discover adapters, preflight
compatibility, run offline verification, understand provider-backed smoke
boundaries, and diagnose failures before trusting a remote model path.

## Goals

- Adapter discovery: expose available adapters with stable metadata: name,
  transport, supported skills, model profile, capability limits, and whether the
  adapter requires provider credentials.
- Compatibility preflight: answer whether a selected adapter can run a selected
  skill on the current host before adapter execution or provider calls begin.
- Provider-backed smoke boundary: add one narrow opt-in provider-backed smoke
  path for a read-only skill, while default tests and acceptance stay offline.
- Cross-OS verification: prove the offline adapter and preflight behavior on
  macOS, Linux, and Windows instead of assuming POSIX-only paths and shells.
- Failure diagnosis: keep unknown adapter, capability mismatch, network policy
  failure, missing credentials, provider failure, and verifier failure separate
  in CLI output, JSON reports, and trace evidence.

## Division of Work

- Adapter boundary owner:
  - define the adapter discovery metadata contract
  - keep `UnsupportedAdapter` for unknown adapter names, unsupported skills, and
    capability mismatch
  - reject provider-specific behavior that bypasses runtime tools, policy, or
    trace
- Runtime and CLI owner:
  - add the smallest discovery and preflight surface needed for agents and CI
  - ensure preflight runs before `adapter.execute`
  - emit machine-readable reports that can be validated by `agenix validate`
- Provider smoke owner:
  - keep the first provider-backed path read-only and explicitly opt-in
  - use stubbed provider tests for default CI
  - record provider metadata in trace without secrets or raw request/response
    bodies
- Cross-platform owner:
  - run the offline suite on macOS, Linux, and Windows
  - cover executable lookup, path normalization, shell policy, artifact
    materialization, and verifier invocation
  - document allowed platform differences instead of hiding them in tests
- Diagnostics owner:
  - keep stable error classes and actionable messages
  - make failed preflight reports useful without requiring a developer to read
    raw trace files first
- Release coordinator:
  - update release notes only after implementation lands
  - keep English and Chinese docs in sync
  - do not mark V0.3 complete until the release gate commands pass

## Acceptance Commands

These are V0.3 release targets. They are not expected to pass merely because
this planning document exists.

Default offline gate:

```bash
go test -count=1 ./...
go vet ./...
go build ./cmd/agenix
go run ./cmd/agenix acceptance --v0.2
go run ./cmd/agenix acceptance --v0.3
```

Adapter discovery target:

```bash
go run ./cmd/agenix adapters
go run ./cmd/agenix adapters --json
```

Compatibility preflight target:

```bash
go run ./cmd/agenix adapters compatible examples/repo.analyze_test_failures.remote/manifest.yaml
go run ./cmd/agenix adapters compatible examples/repo.analyze_test_failures.remote/manifest.yaml --json
```

Current preflight is intentionally metadata-only. It checks adapter skill
support, capability requirements, and remote-network policy before execution.
It does not call providers, execute tools, run verifiers, or mutate workspaces.

Cross-OS gate:

Run the default offline gate on each supported host:

- macOS
- Linux
- Windows

Provider-backed smoke:

```bash
test -n "$OPENAI_API_KEY" && AGENIX_PROVIDER_SMOKE=1 go run ./cmd/agenix run examples/repo.analyze_test_failures.remote/manifest.yaml --adapter openai-analyze
```

The provider-backed smoke is opt-in. It must not be required for default local
tests or default CI. A release can record it as `skipped_no_credentials` only if
the offline adapter, preflight, trace, and diagnostics gates still pass.

## Risks

- Discovery and preflight may grow into a plugin system too early. V0.3 should
  expose only the metadata needed to decide whether a run is safe to start.
- Provider output is nondeterministic. Default verification must use local stub
  tests, not a live provider response.
- Provider credentials can leak through logs, traces, or failure reports. Trace
  metadata must exclude API keys and raw provider payloads.
- Windows behavior can diverge in path handling, executable lookup, shell
  invocation, and artifact materialization. Host-independent tests should cover
  the contract, and real Windows runs should verify the release gate.
- Error classes can collapse under pressure from provider-specific failures.
  Keep top-level classes stable and put provider detail in structured metadata.
- Acceleration can create overclaiming. Release notes must say "target" or
  "planned" until implementation and verification have landed.

## Out of Scope

- marketplace or public registry
- multi-provider abstraction
- provider-side tool calling
- streaming provider output
- default remote execution
- provider-backed write skills
- cloud orchestration

# Agenix v0.3.0 加速计划

V0.3 是 Model Adapter Readiness release。

状态：这是 V0.2.0 之后的活跃目标。第一步 adapter discovery 和 compatibility
preflight 已经落地；其余内容在 V0.3 release gate 存在之前仍然是目标计划。

本次 release 的目标，是让真实 model adapter 可以被安全使用，同时不削弱现有
runtime contract。这里的 "ready" 指开发者能够发现 adapter、提前检查兼容性、运行
离线验收、理解 provider-backed smoke 的边界，并在信任远程模型路径之前诊断失败。

## 目标

- Adapter discovery：暴露可用 adapter 的稳定元数据，包括 name、transport、
  supported skills、model profile、capability limits，以及是否需要 provider
  credentials。
- Compatibility preflight：在 adapter execution 或 provider call 开始之前，回答
  选定 adapter 是否能在当前 host 上运行选定 skill。
- Provider-backed smoke 边界：为 read-only skill 增加一条窄的 opt-in provider
  smoke 路径，同时默认测试和 acceptance 仍然保持离线。
- 跨 OS 验证：在 macOS、Linux、Windows 上证明离线 adapter 和 preflight 行为，而
  不是假设 POSIX path 和 shell 行为。
- 失败诊断：在 CLI output、JSON report 和 trace evidence 中区分 unknown adapter、
  capability mismatch、network policy failure、missing credentials、provider
  failure 和 verifier failure。

## 分工

- Adapter boundary owner：
  - 定义 adapter discovery metadata contract
  - 继续用 `UnsupportedAdapter` 表示 unknown adapter name、unsupported skill 和
    capability mismatch
  - 拒绝绕过 runtime tools、policy 或 trace 的 provider-specific 行为
- Runtime and CLI owner：
  - 增加 agents 和 CI 需要的最小 discovery / preflight surface
  - 确保 preflight 发生在 `adapter.execute` 之前
  - 输出可被 `agenix validate` 校验的 machine-readable report
- Provider smoke owner：
  - 第一条 provider-backed 路径必须保持 read-only，并且显式 opt-in
  - 默认 CI 使用 stubbed provider tests
  - trace 只记录 provider metadata，不记录 secrets 或 raw request/response bodies
- Cross-platform owner：
  - 在 macOS、Linux、Windows 运行 offline suite
  - 覆盖 executable lookup、path normalization、shell policy、artifact
    materialization 和 verifier invocation
  - 文档化允许的平台差异，不把差异藏进测试里
- Diagnostics owner：
  - 保持稳定 error classes 和可行动的错误信息
  - 让失败的 preflight report 足够有用，不要求开发者先读 raw trace files
- Release coordinator：
  - 只在实现落地后更新 release notes
  - 保持英文和中文文档同步
  - release gate commands 通过前，不标记 V0.3 完成

## 验收命令

这些是 V0.3 release targets。不能因为本文档存在，就认为它们现在应该通过。

默认离线 gate：

```bash
go test -count=1 ./...
go vet ./...
go build ./cmd/agenix
go run ./cmd/agenix acceptance --v0.2
go run ./cmd/agenix acceptance --v0.3
```

Adapter discovery target：

```bash
go run ./cmd/agenix adapters
go run ./cmd/agenix adapters --json
```

Compatibility preflight target：

```bash
go run ./cmd/agenix adapters compatible examples/repo.analyze_test_failures.remote/manifest.yaml
go run ./cmd/agenix adapters compatible examples/repo.analyze_test_failures.remote/manifest.yaml --json
```

当前 preflight 有意保持 metadata-only。它在执行前检查 adapter skill support、
capability requirements 和 remote-network policy；不会调用 provider、执行工具、运行
verifier 或修改 workspace。

跨 OS gate：

在每个受支持 host 上运行默认离线 gate：

- macOS
- Linux
- Windows

Provider-backed smoke：

```bash
test -n "$OPENAI_API_KEY" && AGENIX_PROVIDER_SMOKE=1 go run ./cmd/agenix run examples/repo.analyze_test_failures.remote/manifest.yaml --adapter openai-analyze
```

Provider-backed smoke 是 opt-in。它不能成为默认本地测试或默认 CI 的要求。如果
offline adapter、preflight、trace 和 diagnostics gates 都通过，release 可以把它记录
为 `skipped_no_credentials`。

## 风险

- Discovery 和 preflight 可能过早膨胀成 plugin system。V0.3 只应该暴露判断一次
  run 是否可以安全开始所需的 metadata。
- Provider output 有不确定性。默认 verification 必须使用 local stub tests，而不是
  live provider response。
- Provider credentials 可能通过 logs、traces 或 failure reports 泄漏。Trace metadata
  必须排除 API keys 和 raw provider payloads。
- Windows 行为可能在 path handling、executable lookup、shell invocation 和 artifact
  materialization 上分叉。应使用 host-independent tests 覆盖 contract，并用真实
  Windows run 验证 release gate。
- Provider-specific failure 很容易让 error classes 重新混在一起。保持 top-level
  classes 稳定，把 provider detail 放进 structured metadata。
- 加速开发容易导致 overclaiming。Release notes 在实现和验证落地前，只能使用
  "target" 或 "planned"。

## 不在范围内

- marketplace 或 public registry
- multi-provider abstraction
- provider-side tool calling
- streaming provider output
- default remote execution
- provider-backed write skills
- cloud orchestration

# Agenix v0.2.0 Release Notes

V0.2.0 is the Skill Authoring release.

The release moves Agenix from "the canonical demo runs" to "a new skill author
has a repeatable path":

- list built-in authoring templates
- initialize a skill from a template
- validate the generated manifest
- build a portable `.agenix` artifact
- run the artifact through an explicit local adapter
- run `agenix check` as a one-command authoring gate
- validate both passing and failing `check_report` JSON

## Release Gate

Run the V0.2 authoring acceptance gate:

```bash
go run ./cmd/agenix acceptance --v0.2
```

Expected output:

```text
status=passed release=v0.2 templates=2 skills=2 checks=3 failure_reports=1
```

The gate covers:

- `python-pytest` template generation and read-only execution
- `repo-fix-test-failure` template generation and scoped repair execution
- manifest validation for generated skills
- artifact build, inspect, and run for generated skills
- passing `agenix check` reports
- failing `agenix check` reports with `error_class`, `error_message`, trace path,
  and non-zero CLI behavior
- schema validation for generated `check_report` documents

## What Changed

- Added `agenix init templates` and `agenix init templates --json`.
- Added `agenix init skill <name> --template <template> -o <dir>`.
- Added `agenix check <skill-dir|manifest|artifact> [--adapter <name>] [--json]`.
- Added the published `specs/check-report.schema.json` contract.
- Extended `agenix validate` to recognize check reports.
- Added failing-check JSON reports, so CI and other agents get structured
  evidence even when the gate fails.
- Added a beginner tutorial for writing a first skill.

## Boundary

V0.2.0 stays inside the local reference runtime. It does not add a remote
executor, registry marketplace, stronger sandbox, signing, or default
provider-backed model execution. The goal is a reliable authoring loop, not a
larger trust boundary.

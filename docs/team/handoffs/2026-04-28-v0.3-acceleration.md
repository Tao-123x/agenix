## Goals

- Make adapters discoverable before execution.
- Make compatibility preflight explicit before any provider call.
- Keep provider-backed smoke opt-in, read-only, and outside default CI.
- Verify offline behavior on macOS, Linux, and Windows.
- Improve failure diagnostics so agents can recover without reading raw traces
  first.

## Division of Work

- Adapter discovery track:
  - define the minimum adapter metadata needed by humans, agents, and CI
  - expose name, transport, supported skills, model profile, capabilities, and
    provider credential requirements
  - produce JSON that can become a stable validation target
- Compatibility preflight track:
  - check adapter existence, supported skill, capability requirements, transport,
    network policy, host OS assumptions, and executable availability before
    execution
  - fail before provider calls when policy or compatibility is invalid
  - preserve current top-level error classes
- Provider-backed smoke track:
  - keep the first provider-backed path read-only
  - use stubbed provider tests for default verification
  - require explicit credentials and opt-in env for live smoke
  - never persist API keys or raw provider payloads in trace
- Cross-OS track:
  - run the offline gate on macOS, Linux, and Windows
  - capture path, shell, executable lookup, artifact materialization, timeout,
    and verifier differences
  - add contract tests for differences that can be simulated on one host
- Failure diagnostics track:
  - make failed discovery or preflight reports actionable
  - keep CLI messages short, but include the next useful recovery step
  - ensure JSON reports preserve stable `error_class` values
- Release coordination track:
  - keep V0.3 plan docs synchronized in English and Chinese
  - convert target commands into release notes only after they pass
  - call out skipped provider smoke with a concrete reason, not as success

## Acceptance Commands

These are target commands for the implementation work. The default offline
V0.3 gate now exists, and GitHub runs it across Linux, macOS, and Windows.
Remaining tracks should keep it passing while adding diagnostics and explicit
live provider smoke records.

```bash
go test -count=1 ./...
go vet ./...
go build ./cmd/agenix
go run ./cmd/agenix acceptance --v0.2
go run ./cmd/agenix acceptance --v0.3
go run ./cmd/agenix adapters
go run ./cmd/agenix adapters --json
go run ./cmd/agenix adapters compatible examples/repo.analyze_test_failures.remote/manifest.yaml
go run ./cmd/agenix adapters compatible examples/repo.analyze_test_failures.remote/manifest.yaml --json > /tmp/agenix-v0.3-preflight.json
go run ./cmd/agenix validate /tmp/agenix-v0.3-preflight.json
```

Provider-backed live smoke remains explicit:

```bash
go run ./cmd/agenix acceptance --v0.3 --provider-smoke
```

## Risks

- Teams may build a broad adapter plugin system instead of the smallest
  discovery and preflight surface needed for V0.3.
- Provider-backed smoke can become flaky or costly if it enters default CI.
- Missing credential failures can be misclassified as unsupported adapters.
- Windows path and shell behavior can still fail in CI even when macOS local
  verification passes; treat matrix failures as release blockers.
- Faster parallel work can create documentation drift. Release coordination must
  reconcile target docs, implementation, and actual command output before
  release notes are written.

## Customer Alignment

Maya impact:

  path can exist behind policy, trace, and verifier boundaries.
- It should also reduce operational risk by letting users find incompatible
